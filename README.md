# IMJ_UTIL

Утилита для предварительного анализа изображений через GigaChat (мультимодальная модель) с сохранением структурированных отчётов в SQLite. Реализует требования из `ТЗ1.txt`: оценка юридически значимых рисков по 6 категориям с рекомендательными выводами для модераторов.

## Возможности

- Анализ одного изображения по URL
- Пакетный анализ из файла `issues-examples-v2.xls` (колонка `image_url`)
- Сохранение полного JSON-отчёта и исходных полей строки Excel в БД
- Проверка доступности ресурсов GigaChat перед каждым изображением в пакетном режиме
- Досрочное завершение цикла при исчерпании лимитов API
- Просмотр истории анализов из командной строки
- Отдельный read-only web-viewer для базы (не мешает пакетной записи)

## Требования

- Python 3.10+
- Учётные данные GigaChat API (scope `GIGACHAT_API_PERS`) и **пакет токенов GigaChat Pro** для анализа изображений
- Файл `issues-examples-v2.xls` для пакетного режима (в корне проекта или путь через `--issues-file`)

## Установка

### Настройка GigaChat (ключи и оплата)

Перед запуском утилиты нужен проект GigaChat API и ключ авторизации.

**Важно:** для анализа изображений утилита использует мультимодальную модель **`GigaChat-Pro`** (переменная `GIGACHAT_VISION_MODEL` в `.env`). Нужен оплаченный **пакет токенов именно GigaChat Pro** — пакеты Lite, Max или бесплатный freemium-лимит для текстовых моделей **не подходят** для этой утилиты. При покупке в личном кабинете выбирайте пакет с моделью `GigaChat-Pro` / `GigaChat-2-Pro` (см. [тарифы для физлиц](https://developers.sber.ru/docs/ru/gigachat/tariffs/individual-tariffs), для юрлиц — [тарифы B2B](https://developers.sber.ru/docs/ru/gigachat/tariffs/legal-tariffs)). Баланс Pro проверяется перед каждым новым изображением в пакетном режиме; при нулевом остатке анализ останавливается.

## Быстрый запуск (Windows)

В корне проекта три `.bat`-файла — двойной щелчок или запуск из cmd. Все скрипты переходят в папку проекта, используют Python из `.venv` и передают дополнительные аргументы командной строки (`%*`).

| Файл | Назначение |
|---|---|
| `analyze_batch.bat` | Пакетный анализ из `issues-examples-v2.xls` (`main.py analyze-batch`) |
| `viewer.bat` | Web-viewer базы на http://127.0.0.1:8765 (только чтение) |
| `export_risks.bat` | Выгрузка в `data/elevated_risks.csv` записей с риском выше `low` |

```text
analyze_batch.bat
viewer.bat
export_risks.bat
```

Примеры с параметрами (как у соответствующих Python-команд):

```text
analyze_batch.bat --issues-file issues-examples-v2.xls
viewer.bat --port 9000
export_risks.bat -o data\my_export.csv
```

Перед первым запуском нужны установленные зависимости (см. выше) и файл `.env` с ключом GigaChat. `viewer.bat` и `export_risks.bat` можно запускать параллельно с `analyze_batch.bat` — чтение БД не блокирует запись.

## Переменные окружения

| Переменная | Описание |
|---|---|
| `TEST_IMAGE_URL` | URL тестового изображения для `analyze` без `--url` |
| `GIGACHAT_BASIC_AUTH` | Base64-ключ авторизации GigaChat |
| `GIGACHAT_VERIFY` | Проверка SSL (`false` для dev) |
| `GIGACHAT_VISION_MODEL` | Модель для анализа изображений (по умолчанию `GigaChat-Pro`) |
| `GIGACHAT_TIMEOUT` | Таймаут запросов к GigaChat API, сек (по умолчанию `60`) |
| `IMAGE_DOWNLOAD_TIMEOUT` | Таймаут скачивания изображений по URL, сек (по умолчанию `120`) |
| `DATABASE_PATH` | Путь к файлу SQLite (по умолчанию `data/analyses.db`) |

## Команды

### Одно изображение

```powershell
python main.py analyze
python main.py analyze --url "https://example.com/image.jpg"
```

### Пакетный анализ из Excel

```powershell
python main.py analyze-batch
python main.py analyze-batch --issues-file issues-examples-v2.xls --image-field image_url
```

В терминал выводится только общий прогресс:

```
Старт пакетного анализа: 1570 изображений
Прогресс: 1/1570 | ok=1 | cached=0 | errors=0
Прогресс: 2/1570 | ok=2 | cached=1 | errors=0
...
Готово: обработано 1570/1570 | ok=1560 | cached=120 | errors=10
```

Изображения, уже успешно проанализированные ранее (по `image_url` в БД), повторно в GigaChat не отправляются: в прогрессе засчитываются как `ok` и `cached`, в базе остаётся прежняя запись.

При исчерпании ресурсов GigaChat цикл завершается досрочно с сообщением в stderr.

### Просмотр результатов

```powershell
python main.py list
python main.py list --limit 50
python main.py show 42
```

### Web-viewer (только чтение)

Отдельный процесс, открывает БД в режиме `read-only` и не блокирует `analyze-batch`:

```powershell
python viewer.py
python viewer.py --host 127.0.0.1 --port 8765
```

Откройте в браузере: http://127.0.0.1:8765

Страница списка обновляется каждые 15 секунд — удобно смотреть прогресс во время пакетного анализа. Параметры фильтров (статус, уровни риска, страница) сохраняются в URL и не сбрасываются при автообновлении; при возврате с карточки анализа или журнала событий восстанавливается последний открытый список.

На главной странице viewer есть **независимый фильтр по уровню риска** (`none`, `low`, `medium`, `high`, `critical`): отметьте нужные уровни и нажмите «Применить» или ссылку «Все уровни». Фильтр риска работает отдельно от фильтра статуса (успех/ошибки) и сохраняется при переключении страниц.

### Выгрузка рисков в CSV

Вспомогательная утилита выгружает строки с риском **выше `low`** (`medium`, `high`, `critical`):

```powershell
python export_risks.py
python export_risks.py -o data/elevated_risks.csv --issues-file issues-examples-v2.xls
```

В CSV попадают все поля соответствующей строки из `issues-examples-v2.xls`, общий риск, категории с повышенным риском и признаки (`detected_signs`). Кодировка UTF-8 с BOM — для корректного открытия в Excel.

Экспорт открывает БД только на чтение (`?mode=ro`, как viewer) и не мешает `analyze-batch`. Снимок всех строк на один момент времени не гарантируется — при длительной выгрузке в CSV могут попасть записи, сохранённые в БД в разное время.

## База данных

### Таблица `image_analyses`

Хранит результат анализа каждого изображения:

- `image_url`, `overall_risk_level`, `manual_review_required`
- `report_json` — полный отчёт GigaChat
- `raw_response`, `api_usage_json`, `api_response_json`
- `output_json` — полный структурированный выход утилиты (терминал + метаданные)
- `text_on_image`, `confidence`, `recommendations_json`, `risks_json`, `disclaimer`
- `source_file`, `source_row_index`, `source_payload_json` — привязка к строке Excel
- `parsed_from_truncated_json` — `1`, если отчёт восстановлен из обрезанного ответа модели (альтернативный парсер)
- `status` — `success` или `error`

### Таблица `utility_events`

Журнал всех событий утилиты: старт/прогресс/завершение пакета, пропуски из кэша, успехи, ошибки, проверки ресурсов.

### Таблица `gigachat_resource_checks`

Журнал проверок ресурсов GigaChat в пакетном режиме:

- `available`, `exhausted`, `message`, `balance_json`
- `source_file`, `source_row_index` — контекст шага цикла

## Промпт анализа — `imj_util/prompts.py`

**Это главный настраиваемый элемент всей утилиты.** Текст из `ANALYSIS_PROMPT` отправляется в GigaChat при каждом анализе изображения (`gigachat_client.py`). От формулировок в этом файле напрямую зависят:

- уровни риска по шести категориям и `overall_risk_level`;
- признаки в `detected_signs`, которые попадают в БД, viewer и CSV-выгрузку;
- решение о `manual_review_required` и рекомендации модератору;
- доля ложных срабатываний и пропусков при пакетной обработке тысяч изображений.

Изменения в коде парсера, схемы БД или CLI **не заменяют** правки промпта: если модель трактует категории иначе, чем нужно по `ТЗ1.txt`, единственный практичный рычаг — уточнить инструкции здесь. После правки промпта уже сохранённые в БД отчёты **не обновляются** автоматически (срабатывает кэш по `image_url`); для пересчёта нужен повторный анализ или очистка соответствующих записей.

Ниже — **полная копия** текущего содержимого файла `imj_util/prompts.py` (константа `ANALYSIS_PROMPT`):

```
Ты — ИИ-эксперт по предварительной модерации изображений.
Проанализируй изображение и верни ТОЛЬКО валидный JSON без markdown-обёртки.

Оцени риски по категориям из ТЗ:
1. sexual_content — сексуализированное содержание
2. violence_humiliation — насилие и унижение
3. minors — участие несовершеннолетних
4. discrimination_hate — дискриминационные и враждебные элементы
5. portrait_rights — нарушения прав изображённых лиц
6. lgbt_propaganda — признаки пропаганды нетрадиционных отношений, смены пола, педофилии

Для каждой категории укажи:
- risk_level: "none" | "low" | "medium" | "high" | "critical"
- detected_signs: массив кратких описаний найденных признаков (пустой массив, если нет)
- legal_area: затронутая правовая область или null
- description: краткое пояснение оценки

Также верни поля верхнего уровня:
- risks: массив объектов по всем 6 категориям (поле category в каждом объекте)
- text_on_image: распознанный текст на изображении или null
- overall_risk_level: максимальный уровень риска по всем категориям
- recommendations: массив рекомендаций (например, "дополнительная проверка юристом")
- manual_review_required: true/false
- manual_review_reason: причина ручной проверки или null
- confidence: число от 0.0 до 1.0 — уверенность в оценке
- disclaimer: "Выводы носят рекомендательный характер; окончательное решение принимает человек."

Не утверждай законность/незаконность в терминах статей закона.
Опирайся только на видимое содержание изображения.
```

Уровни риска: `none`, `low`, `medium`, `high`, `critical`. Выводы носят рекомендательный характер и не заменяют юридическую экспертизу.

### Рекомендации по внесению уточнений

1. **Сверяйтесь с `ТЗ1.txt`.** Любое изменение формулировок категорий или критериев в промпте должно соответствовать ТЗ; при расхождении сначала обновите ТЗ, затем промпт.

2. **Не ломайте JSON-схему без сопутствующих правок кода.** Имена полей (`risk_level`, `detected_signs`, `risks`, `category`, `overall_risk_level` и т.д.) и допустимые значения уровней риска зашиты в `gigachat_client.py` и `database.py`. Меняйте структуру ответа только вместе с парсером.

3. **Уточняйте пороги, а не только названия категорий.** Если в выгрузке `export_risks.py` слишком много `medium` или, наоборот, пропуски — добавьте в промпт примеры пограничных случаев: «драка в комиксе без крови → low», «ребёнок на фоне без участия в сцене → low или none» и т.п. Конкретные примеры для модели эффективнее общих слов «будь строже».

4. **Разделяйте категории, которые модель путает.** Частая ошибка — завышение `violence_humiliation` и `minors` на одном кадре. Явно опишите, когда повышать каждую категорию отдельно и когда учитывать только доминирующий риск.

5. **Сохраняйте ограничения по праву и видимости.** Строки «не утверждай законность в терминах статей» и «опирайся только на видимое содержание» снижают галлюцинации; не удаляйте их без веской причины.

6. **Проверяйте на малой выборке перед пакетом.** После правки промпта прогоните 10–20 известных изображений:

   ```powershell
   python main.py analyze --url "https://..."
   python main.py show <id>
   python export_risks.py -o data/test_export.csv
   ```

   Сравните `detected_signs` и уровни с ожиданием модератора.

7. **Учитывайте кэш БД.** Успешный анализ по `image_url` в пакетном режиме не пересчитывается. Для проверки нового промпта на уже обработанных URL удалите соответствующие строки из `image_analyses` или используйте тестовые URL.

8. **Фиксируйте изменения в git с пояснением.** В сообщении коммита кратко укажите, какую проблему решало уточнение (например: «снизить ложные medium по minors на фоновых персонажах») — это упростит откат и сравнение выгрузок.

9. **Длинный промпт = больше токенов.** Каждое уточнение увеличивает стоимость пакетной обработки; добавляйте только формулировки, которые реально меняют качество на вашей выборке.

## Ограничения

- Анализ одного изображения занимает несколько секунд (загрузка, upload в GigaChat, запрос к модели)
- Иногда GigaChat обрезает JSON-ответ по лимиту токенов; утилита пытается восстановить полный отчёт из неполного JSON и сохраняет его как `success` с флагом `parsed_from_truncated_json=1` (рекомендуется ручная проверка)
- GigaChat не принимает GIF при загрузке в `/files`; утилита автоматически конвертирует GIF в PNG перед `upload_file` (берётся первый кадр анимации)
- При частых запросах GigaChat может вернуть `429 Too Many Requests` — проверка баланса пропускается, анализ продолжается
- При исчерпании баланса модели `GIGACHAT_VISION_MODEL` (в т.ч. `402 Payment Required`) пакетный режим останавливается **до** анализа следующего изображения. После первой такой ошибки повторные проверки в цикле не выполняются.
- Файл `.env` с ключами не должен попадать в git (см. `.gitignore`)
